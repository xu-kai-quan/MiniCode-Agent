# 01-bash-only — 一个能"自己干活"的最小 Agent (MiniCode Agent v1)

这是一个用本地 Qwen3.5-2B 模型驱动的**编码小助手**。给它一句话, 它会自己规划步骤、调用工具 (执行命令、读写文件)、一步步把活干完。

整个程序就一个文件 [todo.py](todo.py), 大约 370 行 Python。如果你是第一次接触"AI agent"这种东西, 这个项目就是为你准备的: 它把所有概念都摊开放在台面上, 没有任何框架封装。

---

## 一、它能做什么

举个例子, 你这样启动它:

```sh
python todo.py "在 mypkg/ 下建一个包, 包含 __init__.py 和一个 add(a,b) 函数"
```

它会自动:

1. 先思考一下 ("我应该先建目录, 再写两个文件")
2. 调用 `todo` 工具把步骤列出来
3. 调用 `bash` 创建目录
4. 调用 `write_file` 写入 `__init__.py`
5. 调用 `write_file` 写入 `utils.py`
6. 把 todo 项全部标记为完成
7. 退出

整个过程你能在终端里**看见每一步**: 模型在想什么、决定做什么、工具实际跑出了什么。

## 二、核心概念 (零基础友好版)

### 1) 什么叫 "agent"?

普通调用大模型: 你问一句, 它答一句, 结束。

Agent 是这样: 你问一句, 模型答一句**带着工具调用**, 系统真的去执行那个工具, 把结果再喂回模型, 模型看到结果再决定下一步…… 直到模型说"我做完了"。

简单说就是一个 `while` 循环:

```python
while True:
    回复 = 模型(对话历史)
    工具调用列表 = 解析(回复)
    if 没有工具调用:
        break          # 模型不再想干活, 退出
    for 调用 in 工具调用列表:
        结果 = 执行(调用)
        对话历史.append(结果)
```

**这个循环就是 agent 的全部本质**。本项目里, 它在 [todo.py](todo.py) 的 `agent_loop` 函数里, 你能一眼看穿。

### 2) 工具是什么?

工具就是"模型可以请求系统帮它做的事"。本项目内置 5 个:

| 工具名 | 干什么 |
|---|---|
| `bash` | 执行任意 shell 命令 |
| `read_file` | 读取一个文本文件 |
| `write_file` | 创建或覆盖文件 |
| `edit_file` | 把文件里某段文本精确替换成新文本 |
| `todo` | 写/更新一个待办列表 (用来规划步骤) |

模型并不能直接动你的电脑 — 它只能"输出一段表示工具调用的文本", 由系统来真正执行。这是一道很重要的安全边界。

### 3) 为什么要有 todo (规划) 工具?

小模型 (2B 参数) 在多步任务里很容易**走神**: 对话越长, 它越会忘记最初的目标, 或者重复做已经做过的事。

解决办法是让它**先把步骤写下来**, 写下来的东西会留在对话历史里, 之后每一轮模型都能看到, 相当于一份"外部记忆"。规则有两条:

- 任何超过一步的任务, 必须先调 `todo` 列出步骤。
- 同一时刻只能有一个步骤是 `in_progress`, 强制它**一件一件来**。

如果模型连续 3 轮都没更新 todo, 系统会主动注入一条提醒 ("该更新规划了"), 这个机制叫 **nag**。

### 4) 为什么要"沙箱"?

模型理论上可以请求写到任何路径。万一它想 `write_file("C:/Windows/system32/...")` 怎么办?

所以文件类工具都先做一次**路径检查**: 把传入的路径解析成绝对路径, 如果它不在启动时的当前目录之下, 直接拒绝。这样模型的破坏范围就被锁在工作目录里了。

(注意: `bash` 工具是个例外 — 既然叫 bash, 就把权力交给 shell 了, 它能 `cd ..` 出去。这是已知的取舍, 想要更安全的话, 别用 bash。)

### 5) 为什么模型输出里有 `<think>` 又有 `<tool_call>`?

Qwen 这类模型会在正式回答前写一段 `<think>...</think>`, 那是它的内部推理。我们打印它出来给你看 (帮助理解模型在想什么), 但**不存回对话历史** — 否则历史会被前几轮的 think 越塞越重。

`<tool_call>...</tool_call>` 才是它真正请求系统执行的部分, 系统用正则把它抠出来, 找到对应的工具函数执行。

## 三、怎么跑

### 准备工作

1. **下载模型**到本地 (Qwen3.5-2B), 默认放在 `E:/MYSELF/model/qwen/Qwen3.5-2B/`。如果你放在别处, 编辑 [todo.py](todo.py) 顶部的 `MODEL_DIR`。
2. **装依赖**:
   ```sh
   pip install torch transformers safetensors
   ```
3. **有 NVIDIA 显卡更好** (CUDA), 没有也能跑 CPU, 就是慢很多。

### 启动

进到你**希望它工作的目录** (所有文件操作都被锁在启动时的 cwd 里), 然后:

```sh
# 用内置默认任务跑一次
python path/to/v1_mvp/todo.py

# 自己写个任务
python path/to/v1_mvp/todo.py "把 hello.py 改成带类型提示和 main guard 的版本"
```

模型不再调用工具时自动退出, 或者跑满 20 轮强制退出。**没有交互式对话** — 一次只跑一个任务。

### 想调点什么

[todo.py](todo.py) 顶部几行就是全部旋钮:

```python
MODEL_DIR = ...        # 模型路径
WORKDIR = Path.cwd()   # 工作目录 (启动时锁定)
SYSTEM = "..."         # 给模型看的系统提示词
MAX_ROUNDS = 20        # 最多跑几轮就强制退出
MAX_NEW_TOKENS = 1024  # 单轮最多生成多少 token
```

## 四、读源码的建议路线

如果你想搞懂"agent 到底是怎么工作的", 按这个顺序看 [todo.py](todo.py) 最高效:

1. **先看 `agent_loop` 函数** (文件末尾): 这就是骨架。
2. **再看 `TOOL_HANDLERS` 字典**: 工具是怎么登记和分发的。
3. **看任意一个 `run_xxx` 函数**: 一个具体工具长什么样。
4. **看 `parse_tool_calls`**: 模型输出的文本是怎么变成结构化调用的。
5. **看 `TodoManager`**: 规划机制怎么实现, 为什么强制单 in_progress。

整个文件没有用任何 agent 框架 (langchain / autogen 之类), 因此你看到的就是真相 — 没有黑盒。

## 五、已知局限

诚实告知:

- **跑完就退出, 没有上下文**: 不能像 ChatGPT 那样连续对话。
- **Windows 下 `bash` 工具其实是 cmd.exe**, `ls` 之类的 Unix 命令会报"不是内部命令"。
- **写长文件可能被截断**: 单轮 token 预算 1024 不够时, 工具调用会被截掉, 看起来像"完成"了但文件根本没写。
- **小模型偶尔不听规则**: 比如同时把两个 todo 设成 in_progress, 此时系统会报错。
- **bash 不在路径沙箱内**: 它能用 `cd ..` 出工作区。

这些限制都是已知的, 主要是为了保持代码足够小、足够好读。
